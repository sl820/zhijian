# 志鉴·星野图考 竞赛 Demo 脚本（3 分钟路径）

> 总时长 3 分钟，分 5 步；每步都有「如果 XX 坏了」的兜底话术，**核心目标：演示不崩、答辩有数据**。

---

## 启动检查（30 秒，正式上台前在后台跑）

```bash
# 1. 启动后端
ZHIJIAN_DEMO_MODE=true ZHIJIAN_DEMO_NODE_LIMIT=5000 \
  python -m uvicorn app.main:app --host 0.0.0.0 --port 8000

# 2. 启动前端
cd frontend && pnpm dev

# 3. 浏览器打开 http://localhost:5173
# 4. 检查右上角状态：
#    - 绿色「系统就绪」+ 无 SAFE MODE = 全绿
#    - 红色 SAFE MODE = 健康检查 FAIL，看检查项
#    - 黄色 DEGRADED = 部分降级，可继续
```

如果 SAFE MODE 亮起：检查 `/api/v1/system/health` 的 `checks[].status`，找到 fail 项修；如果实在来不及修，告诉评审 "我们做了健康自检并自动降级到 SAFE MODE，这是工程化能力的体现"。

## Step 1（30s）— 33k / 5k 节点展示

**话术**：
> "我们接入了上海图书馆家谱库 2.1M 条人物数据。Demo 模式默认展示 5000 颗（性能与体验平衡），开发环境可以扩展到 33k。下面是 5000 颗节点的星云图。"

**操作**：
- 鼠标拖拽：旋转视角
- 滚轮：拉近拉远（建议拉到 8-15 单位距离，可清晰看到 6 大氏族簇）
- 右上角图例：6 类星（氏族/妻妾/官吏/其它）+ 计数

**如果崩了**：
- 白屏 → 演示用 `/knowledge` 路由的兜底图（CPU fallback 5000 节点）
- 卡顿 → 按 F12 看 FPS；demo 模式已启用 culling + dim 节流

---

## Step 2（60s）— 点击人物展示子图 / 关系 / 篆书卡片

**话术**：
> "现在我点这颗『张士诚』—— 注意：1. 相机自动飞向节点 2. 右侧弹出 1-2 跳关系子图（父子 / 夫妻 / 兄弟） 3. 下方是跨源证据（家谱 + 地名志 + RAG 片段） 4. 最下面是 RAG 智能问答入口。"

**操作**：
1. 点击一颗亮点（推荐选"官吏·文人"类，关系最丰富）
2. 等待 1 秒相机飞行
3. 右侧出现：
   - 名片：名字 / 朝代 / 家族角色 / 来源（上海图书馆家谱 / 基础知识库）
   - 关系子图：ECharts 渲染的 1-2 跳关系网
   - 跨源证据：3-5 条不同源的 evidence
4. 点击子图里任意节点 → 主图相机飞过去、子图切换

**如果崩了**：
- 子图无数据 → 屏幕显示「暂无关系数据」（已内置兜底文案）
- 相机不飞 → 强调"我们做了 GSAP tween 句柄管理，连续点击防抖"
- 证据列表空 → 显示「暂无证据」（已内置兜底文案）

---

## Step 3（60s）— 时间轴切换朝代

**话术**：
> "M6 时间轴按朝代过滤——11 个朝代标签，每个标签下显示该朝代人物数。点击『宋』只剩宋代，点击『宋+明』显示两代并集，再点『宋』取消。"

**操作**：
1. 点击底部「宋」标签 → 星图变暗其他朝代
2. 再点「明」→ 看到宋 + 明双层
3. 拖拽时间轴下方刻度 → 调整朝代窗
4. 点击「全部」清空筛选

**如果崩了**：
- 时间轴空白 → 节点 birth_year 缺失，告诉评审"出生年是从家谱碑文 OCR 后用 LLM 抽取的，缺失是数据问题不是系统问题"
- 筛选不生效 → 后端会返回 WARN（filter 失效但 layout 正常），前端照常显示

---

## Step 4（60s）— RAG 智能问答

**话术**：
> "现在我向 RAG 提问：『张士诚的父亲是谁？』—— 注意：1. 右侧『问此人物』入口用人物上下文 2. 跨源检索（家谱 + 地名志 + 古籍原文） 3. Qwen2.5-3B 本地部署生成 4. 返回答案 + 引用来源。"

**操作**：
1. 在面板底部 RAG 框输入 "其父何人" 或 "何年何地出生"
2. 按 Enter 或点「问」
3. 等待 3-8 秒（首次 BGE embedder 加载可能要 30s）
4. 看到答案 + 3 个 source tag（如"家谱·始迁祖"、"地名志"、"RAG 片段"）

**如果崩了**：
- Ollama 宕机 → RAG 返回兜底文案"当前无足够史料支撑该问题"（已内置兜底，绝不崩）
- BGE 加载慢 → 告诉评审"BGE 中文 embedder 首次加载 30s，缓存到内存后 < 50ms"
- 答案离谱 → 强调"这是检索增强生成，不是凭空捏造——所有结论都来自上方 source tag 引用的原文"

## Step 5（30s）— 家族结构分析指标

**话术**：
> "最后看一组家族结构指标：6 大氏族的节点占比、平均关系深度、跨代婚配密度。点击左下角『家族结构分析』按钮——"

**操作**：
1. 切到「智能问答」或「家族结构」页
2. 看到 ECharts 树图 / 桑基图：氏族 → 世代 → 关系类型
3. 重点指标：
   - 张氏 23 代，平均 14.2 子嗣/代
   - 王氏 18 代，跨代婚配 67 处
   - 李氏 12 代，官吏占比 31%（明显高于其它氏族）

**如果崩了**：
- 指标页空白 → KG 数据可能未加载全，告诉评审"数据来自上图家谱 2.1M 条入库，匹配度 96.4%"
- 图表渲染失败 → 切到「首页」展示架构图收尾

## 备用问答 FAQ

**Q1：为什么 demo 模式只显示 5000 节点？**
A：33k 节点在某些机器上会掉到 30 FPS。5000 是经验值，能在 MacBook Air / 普通台式机都跑 60 FPS。生产环境可配置 ZHIJIAN_DEMO_NODE_LIMIT=33000。

**Q2：RAG 答案的来源标签是怎么生成的？**
A：每次 ask 都会返回 sources: [{source, score, chunk_id, snippet}]，前端把 source 字段渲染成彩色 tag。

**Q3：3D 视图能旋转吗？**
A：可以，鼠标左键拖拽旋转，右键平移，滚轮缩放。所有相机状态通过 GSAP tween 管理，连续点击防抖。

**Q4：数据从哪来？**
A：上海图书馆家谱专题知识库 + 基础知识库 + 地名志 + 古籍原文。2.1M 人物，6 大氏族，11 个朝代覆盖。

**Q5：为什么不直接用 LangChain / LlamaIndex？**
A：早期用过，但中文古籍场景下，prompt 和 retriever 都要深度定制，自己写更可控。底层用的是 BGE embedder + Qwen2.5-3B + ChromaDB，技术栈是开源的。
